**Breaking change.** `WorkerPlugins::AddQuery`, `WorkerPlugins::RemoveQuery`, and `WorkerPlugins::SwitchQuery` no longer materialize the full list of affected resource ids into Ruby memory. The previous services plucked every added or removed resource id so the caller could broadcast them to a frontend cache, which on large workplaces (340k+ users) meant loading tens of megabytes of ids into the Rails process for each call. The services now run the INSERT / DELETE as a single streaming statement and return `{affected_count: N}` instead of `{created: [ids]}` / `{destroyed: [ids]}`. `WorkerPlugins::SwitchQuery` still returns `{mode: :created | :destroyed}` alongside the count, but it now decides which mode to use by inspecting the AddQuery affected count rather than plucking candidate ids up front.

`WorkerPlugins::RemoveQuery` also short-circuits the `resource_id IN (SELECT ... FROM <target_table>)` subquery when the caller passes an unscoped relation (no WHERE, joins, group, having, limit, or offset). The `resource_type = ?` filter alone is enough to pin the DELETE to this workplace's links of the given type, so the subquery — which would otherwise materialize every row of the target model — is dropped. Orphaned links (whose resource row has since been deleted) are now deleted alongside live ones for unscoped calls, matching caller intent ("remove everything matching the query").

Callers must migrate from `result.fetch(:created)` / `result.fetch(:destroyed)` to `result.fetch(:affected_count)`. Broadcast shapes downstream (ApiMaker's `workplace_links_created` / `workplace_links_destroyed` events) should follow and move to a coarse `resource_type`-only payload so frontends refetch their visible rows instead of receiving full id lists.
